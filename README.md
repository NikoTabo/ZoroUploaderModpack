# Мод на раздачу файлов игры через TCP

Для использования мода, необходимо подготовить сервер к раздаче:
* Открыть порт, который нужно указать в конфиге мода
* Подготовить сборку или иные раздаваемые файлы
* Создать папку modpack, и указать путь до неё в конфигурационном файле

# Протокол взаимодействия ZoroUploader (Спецификация)

Данный документ описывает сетевой протокол взаимодействия между сервером обновлений `ZoroUploader` и клиентом (лаунчером). Протокол работает поверх TCP-соединения.

---

## 1. Общие сведения и типы данных

Все числовые значения передаются в сетевом порядке байт (**Big-Endian**).
Поскольку сервер написан на Java и использует потоки `DataInputStream` / `DataOutputStream`, клиенту на другом языке программирования необходимо эмулировать следующие типы данных:

*   **Int (4 байта)**: Знаковое 32-битное целое число.
*   **UTF-String**: Строка в модифицированной кодировке UTF-8. Формат передачи:
    1. Первые 2 байта (Unsigned Short) — длина строки в байтах.
    2. Последующие байты — сама строка.
*   **Raw Bytes**: Непрерывный поток байт файла без дополнительных заголовков и метаданных.

---

## 2. Фаза 1: Подключение и передача Манифеста

Сразу после установления TCP-соединения сервер отправляет клиенту манифест в формате JSON. Клиент обязан считать эти данные перед отправкой каких-либо команд.

### Схема инициализации:
1. **Клиент** подключается к серверу по TCP (порт настраивается на сервере).
2. **Сервер** отправляет 4-байтовое число `Int` — длину JSON-строки манифеста в байтах.
3. **Сервер** отправляет массив байт (`Raw Bytes`) указанной длины. Это сериализованный JSON-манифест в кодировке UTF-8.
4. **Клиент** принимает данные, десериализует JSON и использует его для сверки локальных файлов.

> **Примечание:** Манифест содержит информацию о файлах сборки (пути, размеры в байтах и хэш-суммы). Данные о размерах файлов критически важны для последующего скачивания.

---

## 3. Фаза 2: Цикл обработки команд

После получения манифеста соединение переходит в режим ожидания команд от клиента. Каждая команда инициируется клиентом и отправляется в формате `UTF-String`.

### Команда `~ping` (Проверка связи)

Используется для поддержания соединения (Keep-Alive) или проверки его доступности.

*   **Запрос от клиента:** Строка `~ping` (тип `UTF-String`).
*   **Ответ сервера:** Строка `~pong` (тип `UTF-String`).

---

### Команда `~get_modpack` (Загрузка файлов)

Используется для скачивания файлов сборки (модпака).

**Важное ограничение архитектуры сервера:**  
Сервер передает файлы сплошным бинарным потоком. В самом потоке **нет** разделителей, заголовков имен или указания размеров отдельных файлов.  
Клиент **обязан** знать точный размер каждого запрашиваемого файла из манифеста (полученного на Фазе 1), чтобы считывать из сокета ровно нужное количество байт для каждого файла. Если клиент прочитает хотя бы на один байт больше или меньше, весь последующий поток данных будет испорчен.

#### Последовательность шагов:

1. **Клиент** отправляет строку `~get_modpack` (тип `UTF-String`).
2. **Клиент** отправляет `Int` (4 байта) — количество запрашиваемых файлов `M`.
3. **Клиент** последовательно отправляет `M` строк (тип `UTF-String`) — относительные пути к файлам относительно папки `modpack` (например, `mods/Jei.jar`).
4. **Сервер** последовательно отправляет содержимое каждого найденного файла в виде `Raw Bytes`.
5. **Клиент** читает поток байт, разбивая его на отдельные файлы на основе размеров, известных из манифеста.

*Пример:* Если клиент запросил `file1.jar` (100 байт) и `file2.jar` (250 байт), сервер отправит ровно 350 байт. Клиент должен прочитать сначала 100 байт и сохранить как `file1.jar`, а затем следующие 250 байт и сохранить как `file2.jar`.

---

## 4. Завершение сессии и обработка ошибок

*   **Закрытие соединения:** Соединение закрывается по инициативе любой из сторон. Если клиенту больше не требуются операции, он может просто закрыть TCP-сокет.
*   **Неизвестные команды:** Если клиент отправляет строку, не соответствующую спецификации (не `~ping` и не `~get_modpack`), сервер немедленно принудительно закрывает соединение.
*   **Сбои при передаче:** В случае сетевых ошибок чтения или записи на стороне сервера, соединение закрывается. Клиенту рекомендуется реализовать механизм повторного подключения.
